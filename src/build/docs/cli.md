# CRB CLI — Developer Reference

The `crb` CLI is a development orchestration tool shipped inside `@zengenti/contensis-react-base`. Consumer projects invoke it to start the full SSR dev server pipeline (webpack-dev-server + nodemon + HMR) with a single command.

## Architecture

```
consumer project (react-starter)
├── node_modules/
│   ├── @zengenti/contensis-react-base/
│   │   └── cjs/
│   │       ├── build.js              ← CLI binary (bin: crb)
│   │       └── dev-server-globals.js ← standalone -r preload for nodemon
│   ├── webpack/                      ← resolved at runtime by CRB (externalised)
│   ├── webpack-dev-server/           ← resolved at runtime by CRB (externalised)
│   ├── nodemon/                      ← resolved at runtime by CRB (externalised)
│   ├── cac/                          ← resolved at runtime by CRB (externalised)
│   ├── dotenv/                       ← resolved at runtime by CRB (externalised)
│   └── .bin/
│       └── crb → ../contensis-react-base/cjs/build.js
├── webpack/webpack.config.js         ← consumer's dual-target config
├── webpack/define-config.js          ← consumer's runtime defines
├── .env                              ← consumer's env vars
└── package.json
    └── "devDependencies": { "webpack", "webpack-dev-server", "nodemon", ... }
```

The CRB package provides **orchestration logic only**. All runtime dependencies (`webpack`, `webpack-dev-server`, `nodemon`, `cac`, `dotenv`) are **externalised** in the rollup bundle and resolved from the consumer's `node_modules` at runtime.

## Source Files

| File | Role |
|---|---|
| `cli.ts` | Entry point. Uses `cac` for CLI parsing. Defines the `dev` subcommand and delegates to `runDev()`. |
| `dev-server.ts` | Orchestration core. Validates config, loads `.env`, creates dual webpack compilers, starts webpack-dev-server, watches the server compiler, launches nodemon when both compilers are ready, and handles graceful shutdown. |
| `dev-server-globals-runtime.ts` | Standalone runtime file (emitted separately by rollup). Loaded by nodemon via `-r`. Resolves the consumer's `webpack/define-config` at runtime using `process.cwd()` and injects the `development` defines as `global` variables. |

## Rollup Integration

The CLI is bundled by the **existing** `rollup.config.lib.js` — no separate config is needed.

### Input entries

```js
input: {
  // ... existing entries ...
  'build': './src/build/cli',                        → cjs/build.js
  'dev-server-globals': './src/build/dev-server-globals-runtime',  → cjs/dev-server-globals.js
},
```

- `build` entry bundles `cli.ts → dev-server.ts` into `cjs/build.js`
- `dev-server-globals` entry emits a standalone `cjs/dev-server-globals.js` for nodemon's `-r` flag

### Externalised dependencies

These packages are **not** installed in the CRB package. They are externalised so the compiled bundle contains bare `require()` calls that resolve from the consumer's `node_modules`:

```js
external: [
  // ... existing externals ...
  'cac',
  'dotenv',
  'nodemon',
  'webpack',
  'webpack-dev-server',
],
```

### Package.json `bin` entry

```json
"bin": {
  "crb": "./cjs/build.js"
}
```

After `npm run build`, `cjs/build.js` exists with a `#!/usr/bin/env node` shebang. npm/yarn creates the `crb` symlink in `.bin/` so consumers can invoke it directly.

## Type Dependencies

The following devDependencies are needed for TypeScript type-checking only (not runtime):

- `cac` — ships its own types (added as devDependency)
- `@types/webpack` — webpack type definitions
- `@types/webpack-dev-server` — webpack-dev-server type definitions

`dotenv` ships its own types. `nodemon` has no type definitions — uses `// @ts-ignore` on the import.

## Build Output

| File | Purpose |
|---|---|
| `cjs/build.js` | CLI binary. Bundles all orchestration logic. Has `#!/usr/bin/env node` shebang. |
| `cjs/dev-server-globals.js` | Standalone preload file. Injects consumer's webpack defines as globals. |
| `esm/build.js` | ESM byproduct (unused by CLI, generated by dual-output rollup). |
| `models/build/*.d.ts` | TypeScript declarations. |

## Design Decisions

### Why externalise runtime deps?

The consumer project already has its own versions of `webpack`, `webpack-dev-server`, etc. as devDependencies. Externalising avoids version conflicts and keeps the CRB package lightweight. The consumer controls which versions to use.

### Why is `dev-server-globals.js` a separate file?

Nodemon spawns a **new child process** for the SSR server. The `-r` flag preloads a module before the script runs. The preload must be a separate file because:

1. The consumer's `webpack/define-config` path is dynamic (`process.cwd()` + `webpack/define-config`) — rollup cannot bundle it statically
2. The globals must be set in the **child process's** `global` scope, not the CLI process
3. Passing the path string from the bundle lets nodemon resolve and preload the file in its own process

### Why not use a MultiCompiler?

`webpack([config1, config2])` returns a `MultiCompiler` which doesn't expose `compiler.hooks.afterEmit` the same way. The CLI creates two separate `Compiler` instances — one for client, one for server — and coordinates them with state flags.
